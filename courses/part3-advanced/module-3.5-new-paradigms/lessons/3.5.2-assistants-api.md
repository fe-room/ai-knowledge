# 3.5.2 OpenAI Assistants API

> 理解托管式有状态Agent的设计范式

## 一、从自建Agent到托管Agent

### 1.1 传统Agent的状态管理负担

```
自建Agent需要开发者管理的一切：

┌──────────────────────────────────────────────┐
│                 你的代码                      │
│                                              │
│  ┌────────────┐  消息历史                     │
│  │ messages[] │  → 自己管理列表               │
│  └────────────┘  → 自己处理Token超限截断      │
│                  → 自己做持久化存储            │
│                                              │
│  ┌────────────┐  文件处理                     │
│  │  files     │  → 自己上传/解析文件          │
│  └────────────┘  → 自己管理文件存储           │
│                  → 自己做文件清理             │
│                                              │
│  ┌────────────┐  工具执行                     │
│  │  tools     │  → 自己实现代码解释器         │
│  └────────────┘  → 自己实现文件检索           │
│                  → 自己处理工具调用循环        │
│                                              │
│  ┌────────────┐  运行管理                     │
│  │  runs      │  → 自己实现异步执行           │
│  └────────────┘  → 自己处理超时/取消          │
│                  → 自己管理并发               │
└──────────────────────────────────────────────┘

问题：大量样板代码，重复造轮子
```

### 1.2 Assistants API 的思路

```
Assistants API —— 平台托管核心能力：

┌──────────────────────────────────────────────┐
│               你的代码                        │
│  → 定义Assistant的角色和指令                  │
│  → 创建Thread，发送消息                       │
│  → 等待Run完成，获取结果                      │
│                                              │
│  （其余全部交给OpenAI平台）                    │
└──────────────────────────────────────────────┘
        │
        ↓ API调用
┌──────────────────────────────────────────────┐
│             OpenAI 平台托管                    │
│                                              │
│  ✅ 对话历史 → 自动存储在Thread中             │
│  ✅ Token管理 → 自动截断和上下文窗口管理      │
│  ✅ 文件处理 → 内置解析PDF/CSV/代码文件       │
│  ✅ 代码执行 → 内置Code Interpreter沙箱      │
│  ✅ 文件检索 → 内置向量化检索（类似RAG）      │
│  ✅ 运行管理 → 异步执行，可轮询/流式获取      │
└──────────────────────────────────────────────┘
```

## 二、核心概念

### 2.1 四大对象

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  Assistant（助手）                               │
│  ├── 定义：角色、指令、模型、工具               │
│  ├── 生命周期：长期存在，可复用                  │
│  └── 类比：一个配置好的"员工"                    │
│                                                 │
│  Thread（对话线程）                              │
│  ├── 定义：一个独立的对话会话                    │
│  ├── 包含：有序的消息列表                        │
│  ├── 生命周期：跨请求持久化                      │
│  └── 类比：一个持续的对话窗口                    │
│                                                 │
│  Message（消息）                                 │
│  ├── 定义：Thread中的一条消息                    │
│  ├── 角色：user 或 assistant                    │
│  ├── 内容：文本、图片、文件附件                  │
│  └── 类比：对话中的一句话                        │
│                                                 │
│  Run（运行）                                    │
│  ├── 定义：在Thread上执行一次Assistant推理       │
│  ├── 状态：queued → in_progress → completed     │
│  ├── 可能触发：工具调用、代码执行                │
│  └── 类比：一次"请回答"的指令                    │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 2.2 运行流程

```
创建助手 (一次性)
    │
    ↓
┌──────────────────────────────────────────────┐
│  对话循环                                     │
│                                              │
│  创建Thread ──→ 添加Message ──→ 创建Run      │
│                                    │         │
│                              ┌─────↓─────┐   │
│                              │  queued    │   │
│                              └─────┬─────┘   │
│                                    ↓         │
│                              ┌───────────┐   │
│                              │in_progress│   │
│                              └─────┬─────┘   │
│                                    ↓         │
│                          ┌─────────────────┐ │
│                          │ requires_action │ │ ← 需要调用自定义工具？
│                          └────────┬────────┘ │
│                                   ↓          │
│                          提交工具结果          │
│                                   ↓          │
│                              ┌──────────┐    │
│                              │completed │    │
│                              └─────┬────┘    │
│                                    ↓         │
│                            获取Assistant消息  │
└──────────────────────────────────────────────┘
```

## 三、实战开发

### 3.1 创建和使用Assistant

```python
"""
Assistants API 完整使用示例
"""

from openai import OpenAI
import time

client = OpenAI()

# ===== 1. 创建Assistant =====

assistant = client.beta.assistants.create(
    name="数据分析助手",
    instructions="""你是一个专业的数据分析师。
    - 当用户上传数据文件时，使用Code Interpreter分析数据
    - 当用户提问时，先查询知识库获取相关信息
    - 回答要包含具体数据和可视化图表""",
    model="gpt-4o",
    tools=[
        {"type": "code_interpreter"},   # 内置：代码执行沙箱
        {"type": "file_search"},        # 内置：文件检索（RAG）
    ],
)
print(f"Assistant ID: {assistant.id}")
# 保存ID，下次可直接 client.beta.assistants.retrieve(id) 复用


# ===== 2. 创建Thread（对话线程） =====

thread = client.beta.threads.create()
print(f"Thread ID: {thread.id}")


# ===== 3. 发送消息 =====

message = client.beta.threads.messages.create(
    thread_id=thread.id,
    role="user",
    content="帮我分析一下，Python和JavaScript在AI开发中各自的优势是什么？"
)


# ===== 4. 创建Run并等待完成 =====

run = client.beta.threads.runs.create(
    thread_id=thread.id,
    assistant_id=assistant.id,
)

# 轮询等待完成
while True:
    run = client.beta.threads.runs.retrieve(
        thread_id=thread.id,
        run_id=run.id,
    )
    if run.status == "completed":
        break
    elif run.status == "failed":
        print(f"Run失败: {run.last_error}")
        break
    elif run.status == "requires_action":
        # 需要处理自定义工具调用（见3.3节）
        break
    time.sleep(1)


# ===== 5. 获取回复 =====

messages = client.beta.threads.messages.list(thread_id=thread.id)
for msg in messages.data:
    if msg.role == "assistant":
        for block in msg.content:
            if block.type == "text":
                print(block.text.value)
        break  # 最新一条assistant消息
```

```python
Assistants API 内置文件检索 —— 无需自建RAG Pipeline

# ===== 1. 创建向量存储 =====

vector_store = client.beta.vector_stores.create(

    file = client.files.create(file=f, purpose="assistants")

client.beta.vector_stores.files.create(
    vector_store_id=vector_store.id,
    file_id=file.id,
)

# 批量上传
file_paths = ["faq.md", "api_docs.pdf", "changelog.txt"]
file_streams = [open(path, "rb") for path in file_paths]

client.beta.vector_stores.file_batches.upload_and_poll(
    vector_store_id=vector_store.id,
    files=file_streams,
)

for stream in file_streams:
    stream.close()


# ===== 3. 关联到Assistant =====

client.beta.assistants.update(
    assistant_id=assistant.id,
    tool_resources={
        "file_search": {
            "vector_store_ids": [vector_store.id]
        }
    }
)

# 现在对话时，Assistant会自动从上传的文件中检索相关内容
# 无需自己实现：分块、向量化、检索、重排序
```

### 3.3 自定义工具（Function Calling）

```python
"""
Assistants API 中使用自定义工具
当内置工具不够时，可以定义自己的函数
"""

import json

# 创建带自定义工具的Assistant
assistant = client.beta.assistants.create(
    name="全能助手",
    instructions="你可以查询天气、搜索产品",
    model="gpt-4o",
    tools=[
        {"type": "code_interpreter"},
        {
            "type": "function",
            "function": {
                "name": "get_weather",
                "description": "获取城市天气",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "city": {"type": "string", "description": "城市名"},
                    },
                    "required": ["city"],
                }
            }
        },
        {
            "type": "function",
            "function": {
                "name": "search_products",
                "description": "搜索产品信息",
                "parameters": {
                    "type": "object",
                    "properties": {
                        "keyword": {"type": "string"},
                        "category": {"type": "string"},
                    },
                    "required": ["keyword"],
                }
            }
        },
    ]
)


def handle_tool_calls(run, thread_id):
    """处理自定义工具调用"""
    tool_outputs = []

    for tool_call in run.required_action.submit_tool_outputs.tool_calls:
        name = tool_call.function.name
        args = json.loads(tool_call.function.arguments)

        if name == "get_weather":
            # 调用天气服务
            result = {"city": args["city"], "temp": 22, "condition": "多云"}
        elif name == "search_products":
            # 调用产品搜索
            result = {"products": [{"name": "AI开发板", "price": 599}]}
        else:
            result = {"error": f"未知工具: {name}"}

        tool_outputs.append({
            "tool_call_id": tool_call.id,
            "output": json.dumps(result, ensure_ascii=False),
        })

    # 提交结果，Run继续执行
    return client.beta.threads.runs.submit_tool_outputs(
        thread_id=thread_id,
        run_id=run.id,
        tool_outputs=tool_outputs,
    )


# 完整的对话循环（含工具调用处理）
def chat_loop(assistant_id, thread_id, user_message):
    """完整对话循环"""
    client.beta.threads.messages.create(
        thread_id=thread_id,
        role="user",
        content=user_message,
    )

    run = client.beta.threads.runs.create(
        thread_id=thread_id,
        assistant_id=assistant_id,
    )

    while True:
        run = client.beta.threads.runs.retrieve(
            thread_id=thread_id,
            run_id=run.id,
        )

        if run.status == "completed":
            break
        elif run.status == "requires_action":
            run = handle_tool_calls(run, thread_id)
        elif run.status in ("failed", "cancelled", "expired"):
            raise RuntimeError(f"Run {run.status}: {run.last_error}")
        else:
            time.sleep(0.5)

    messages = client.beta.threads.messages.list(thread_id=thread_id, limit=1)
    return messages.data[0].content[0].text.value
```

### 3.4 流式输出

```python
"""
Assistants API 流式输出 —— 实时显示回复
"""

from openai import AssistantEventHandler

class StreamHandler(AssistantEventHandler):
    """流式事件处理器"""

    def on_text_created(self, text):
        print("\nassistant > ", end="", flush=True)

    def on_text_delta(self, delta, snapshot):
        print(delta.value, end="", flush=True)

    def on_tool_call_created(self, tool_call):
        print(f"\n[调用工具: {tool_call.type}]", flush=True)

    def on_tool_call_delta(self, delta, snapshot):
        # Code Interpreter 输出代码
        if delta.type == "code_interpreter" and delta.code_interpreter:
            if delta.code_interpreter.input:
                print(delta.code_interpreter.input, end="", flush=True)


# 使用流式输出
with client.beta.threads.runs.stream(
    thread_id=thread.id,
    assistant_id=assistant.id,
    event_handler=StreamHandler(),
) as stream:
    stream.until_done()
```

## 四、Assistants API vs 自建Agent

### 4.1 能力对比

```
┌─────────────────┬──────────────────────┬──────────────────────┐
│     能力         │  Assistants API      │  自建Agent            │
├─────────────────┼──────────────────────┼──────────────────────┤
│ 对话历史管理     │ ✅ 自动（Thread）     │ 🔧 手动维护messages  │
│ 文件检索/RAG    │ ✅ 内置file_search   │ 🔧 自建Pipeline      │
│ 代码执行        │ ✅ 内置沙箱          │ 🔧 自建或用第三方     │
│ Token管理       │ ✅ 自动截断          │ 🔧 手动计算和处理     │
│ 异步执行        │ ✅ 原生支持          │ 🔧 自己实现           │
│ 模型选择        │ ❌ 仅OpenAI          │ ✅ 任意模型           │
│ 数据控制        │ ❌ 数据在OpenAI      │ ✅ 完全本地/私有      │
│ 检索策略定制    │ ❌ 黑盒              │ ✅ 完全可控           │
│ 成本透明度      │ ⚠️ 隐含存储费用     │ ✅ 完全可控           │
│ 框架灵活性      │ ❌ 绑定OpenAI生态    │ ✅ 自由组合           │
│ 延迟控制        │ ⚠️ 平台调度         │ ✅ 可优化             │
└─────────────────┴──────────────────────┴──────────────────────┘
```

### 4.2 选型建议

```
选择 Assistants API 当：
  ✅ 快速原型开发 / MVP验证
  ✅ 需要代码执行能力但不想自建沙箱
  ✅ 文档问答场景（内置RAG足够用）
  ✅ 团队无AI工程经验，需要平台托管
  ✅ 对数据隐私要求不严格

选择自建Agent 当：
  ✅ 需要多模型支持（Claude / Gemini / 开源模型）
  ✅ 数据隐私要求高（如医疗、金融行业）
  ✅ 需要精细控制检索策略和排序算法
  ✅ 需要深度定制Agent行为和工作流
  ✅ 需要控制成本（Assistants API的存储费用可能很高）
  ✅ 生产级系统的长期维护
```

## 五、JavaScript/TypeScript 实现

```typescript
/**
 * Assistants API TypeScript 示例
 */

import OpenAI from "openai";

const client = new OpenAI();

async function runAssistant() {
  // 创建Assistant
  const assistant = await client.beta.assistants.create({
    name: "代码导师",
    instructions: "你是编程导师，帮助用户理解和编写代码。使用Code Interpreter执行示例。",
    model: "gpt-4o",
    tools: [{ type: "code_interpreter" }],
  });

  // 创建Thread
  const thread = await client.beta.threads.create();

  // 发送消息
  await client.beta.threads.messages.create(thread.id, {
    role: "user",
    content: "用Python实现一个简单的快速排序，并测试几组数据",
  });

  // 流式运行
  const stream = client.beta.threads.runs.stream(thread.id, {
    assistant_id: assistant.id,
  });

  for await (const event of stream) {
    if (event.event === "thread.message.delta") {
      const delta = event.data.delta;
      if (delta.content) {
        for (const block of delta.content) {
          if (block.type === "text" && block.text?.value) {
            process.stdout.write(block.text.value);
          }
        }
      }
    }
  }

  // 清理（生产环境中根据业务需要决定是否删除）
  await client.beta.assistants.del(assistant.id);
}

runAssistant().catch(console.error);
```

## 六、生产注意事项

### 6.1 成本控制

```
Assistants API 隐含成本：

1. Token消耗
   - 每次Run会消耗 instructions + 对话历史 + 检索内容 的Token
   - 长对话的Thread中，每次Run的成本会递增

2. 存储费用
   - Vector Store存储费: $0.10/GB/天（截至2025年初）
   - 文件存储费: 按量计费
   - 长期累积可能超过自建方案

3. Code Interpreter
   - 每次Session有计算时间费用
   - 复杂计算任务成本较高

建议：
  → 定期清理不再使用的Thread和Vector Store
  → 监控每个Assistant的API调用量和费用
  → 评估长期成本是否低于自建方案
```

### 6.2 错误处理模式

```python
"""
生产级错误处理
"""

import time
from openai import APIError, RateLimitError

def robust_run(assistant_id: str, thread_id: str, max_retries: int = 3):
    """带重试的Run执行"""

    for attempt in range(max_retries):
        try:
            run = client.beta.threads.runs.create(
                thread_id=thread_id,
                assistant_id=assistant_id,
            )

            timeout = 120  # 最大等待秒数
            start = time.time()

            while time.time() - start < timeout:
                run = client.beta.threads.runs.retrieve(
                    thread_id=thread_id, run_id=run.id
                )

                if run.status == "completed":
                    return run
                elif run.status in ("failed", "expired"):
                    raise RuntimeError(
                        f"Run {run.status}: {run.last_error}"
                    )
                elif run.status == "requires_action":
                    # 处理工具调用...
                    pass

                time.sleep(1)

            # 超时 → 取消Run
            client.beta.threads.runs.cancel(
                thread_id=thread_id, run_id=run.id
            )
            raise TimeoutError("Run执行超时")

        except RateLimitError:
            wait = 2 ** attempt
            print(f"速率限制，{wait}秒后重试...")
            time.sleep(wait)
        except APIError as e:
            if attempt == max_retries - 1:
                raise
            print(f"API错误: {e}，重试中...")
            time.sleep(1)

    raise RuntimeError("达到最大重试次数")
```

## 七、小结

```
Assistants API 要点回顾：

┌───────────────────────────────────────────────┐
│  1. 核心价值                                   │
│     → 将Agent状态管理托管给平台                 │
│     → 内置RAG和代码执行，降低开发门槛           │
│                                               │
│  2. 四大对象                                   │
│     → Assistant（角色定义）                     │
│     → Thread（对话上下文）                      │
│     → Message（消息内容）                       │
│     → Run（执行推理）                           │
│                                               │
│  3. 适合                                       │
│     → 快速原型、文档问答、代码执行场景          │
│     → 不想自建RAG/状态管理的团队                │
│                                               │
│  4. 不适合                                     │
│     → 多模型、高隐私、精细控制场景              │
│     → 成本敏感的大规模生产系统                  │
│                                               │
│  5. 与自建Agent不是替代关系                     │
│     → 可以混合使用：原型用Assistants，           │
│       生产用自建方案                            │
└───────────────────────────────────────────────┘
```
