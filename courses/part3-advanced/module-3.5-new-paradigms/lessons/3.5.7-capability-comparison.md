# 3.5.7 能力集成机制对比：Function Calling / Tools / MCP / Skills

> 厘清 AI Agent 开发中最容易混淆的四个概念

## 一、为什么需要这一节

```
场景：你是一个Agent开发者，想给Agent添加"查天气"的能力。

你可能看到的术语：
  - "定义一个 Function Calling"
  - "注册一个 Tool"
  - "接入一个 MCP Server"
  - "配置一个 Skill"

它们都能实现"查天气"，但到底有什么区别？

答案：它们处于不同的抽象层次

                层级越高 → 越完整、越自动
                ┌─────────────────────────────── Skill
                │   ┌───────────────────────────── MCP
                │   │   ┌─────────────────────────── Tool
                │   │   │   ┌───────────────────────── Function Calling
                ↓   ↓   ↓   ↓
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  Function Calling ── LLM与外部代码交互的底层通信机制     │
  │       ↑                                                  │
  │  Tool ──────────── 一个具体的、可被调用的功能单元        │
  │       ↑                                                  │
  │  MCP ──────────── 工具的标准化发布与发现协议             │
  │       ↑                                                  │
  │  Skill ────────── 包含上下文的可组合能力单元             │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

---

## 二、逐一定义

### 2.1 Function Calling（函数调用）

```
本质：LLM 的一种输出格式

普通对话：
  用户: "北京天气怎么样？"
  LLM输出: "北京今天晴，25°C"  ← 纯文本，可能是编造的

Function Calling：
  用户: "北京天气怎么样？"
  LLM输出: {"name": "get_weather", "arguments": {"city": "北京"}}
  ← 不是直接回答，而是告诉你"应该调用哪个函数"

关键：
  ✓ Function Calling 是 LLM 的能力（模型层面）
  ✗ 它本身不执行函数 —— 只是生成调用指令
  ✗ 它不知道函数在哪、怎么运行
```

```python
"""
Function Calling 原始用法 —— 最底层的协议
"""

from openai import OpenAI
import json

client = OpenAI()

# 1. 告诉LLM有哪些函数可用（JSON Schema 描述）
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取指定城市的天气信息",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "城市名称"
                }
            },
            "required": ["city"]
        }
    }
}]

# 2. LLM 决定是否调用
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
    tools=tools,
)

# 3. 解析LLM的"调用指令"
tool_call = response.choices[0].message.tool_calls[0]
function_name = tool_call.function.name        # "get_weather"
arguments = json.loads(tool_call.function.arguments)  # {"city": "北京"}

# 4. 你自己负责执行（LLM不管这一步）
result = get_weather(**arguments)  # 你的代码

# 5. 把结果喂回LLM
follow_up = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "user", "content": "北京天气怎么样？"},
        response.choices[0].message,           # LLM的调用指令
        {
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": json.dumps(result),      # 执行结果
        },
    ],
    tools=tools,
)

# 6. LLM 基于函数结果生成最终回答
print(follow_up.choices[0].message.content)
# → "北京今天晴，气温25°C，适合出行。"
```

```
小结：
  Function Calling = LLM的一种结构化输出机制
  角色 = 信使（传递"调谁、传什么参数"的指令）
  不涉及 = 函数如何实现、在哪里运行、怎么发现
```

### 2.2 Tool（工具）

```
本质：一个封装好的、可被Agent调用的功能单元

与 Function Calling 的区别：
  Function Calling = LLM说"我要调get_weather"   （指令层）
  Tool            = get_weather的具体实现代码     （实现层）

Tool = 函数实现 + 元数据描述 + 参数验证 + 错误处理

┌──────────────────────────────────────────┐
│               Tool: get_weather          │
│                                          │
│  描述: "获取指定城市的天气信息"           │
│  参数: city (string, 必填)               │
│  实现: 调用OpenWeatherMap API            │
│  验证: city不能为空                      │
│  错误处理: API超时返回默认提示            │
│                                          │
│  → 通过 Function Calling 暴露给LLM      │
│  → 执行结果返回给Agent                   │
└──────────────────────────────────────────┘
```

```python
"""
Tool —— 框架层面的封装
"""

# === LangChain 中的 Tool 定义 ===

from langchain_core.tools import tool


@tool
def get_weather(city: str) -> str:
    """获取指定城市的天气信息

    Args:
        city: 城市名称，如"北京"、"上海"
    """
    # 实际调用天气API（这就是Tool的"实现"部分）
    import httpx
    resp = httpx.get(
        "https://api.weatherapi.com/v1/current.json",
        params={"key": "YOUR_KEY", "q": city}
    )
    data = resp.json()
    return f"{city}: {data['current']['condition']['text']}, {data['current']['temp_c']}°C"


# LangChain 自动从函数签名 + docstring 生成 Function Calling 的 JSON Schema
print(get_weather.name)        # "get_weather"
print(get_weather.description) # "获取指定城市的天气信息"
print(get_weather.args_schema.model_json_schema())
# → {"properties": {"city": {"type": "string", ...}}, "required": ["city"]}

# 绑定到Agent
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent

llm = ChatOpenAI(model="gpt-4o")
agent = create_react_agent(llm, tools=[get_weather])
# Agent自动：收到问题 → LLM生成Function Calling → 框架执行Tool → 结果送回LLM
```

```typescript
/**
 * Tool —— TypeScript/Vercel AI SDK 中的定义
 */
import { tool } from 'ai';
import { z } from 'zod';

const weatherTool = tool({
  description: '获取指定城市的天气信息',
  parameters: z.object({
    city: z.string().describe('城市名称'),
  }),
  execute: async ({ city }) => {
    const resp = await fetch(
      `https://api.weatherapi.com/v1/current.json?key=YOUR_KEY&q=${city}`
    );
    const data = await resp.json();
    return `${city}: ${data.current.condition.text}, ${data.current.temp_c}°C`;
  },
});
```

```
小结：
  Tool = Function Calling + 具体实现 + 元数据
  角色 = 执行者（真正干活的代码）
  绑定关系 = 一个Tool通常对应一个Function Calling定义
  局限 = 与框架耦合（LangChain的Tool ≠ CrewAI的Tool）
```

### 2.3 MCP（Model Context Protocol）

```
本质：工具的标准化发布与发现协议

与 Tool 的区别：
  Tool = 你在代码里定义的函数     （写在你的项目里）
  MCP  = 工具以独立服务运行       （通过标准协议访问）

类比：
  Tool    = 你电脑里安装的本地软件
  MCP     = 通过USB/网络接入的外部设备（标准接口）

┌────────────────────┐         ┌────────────────────┐
│    MCP Client      │         │    MCP Server      │
│  (Agent/IDE/App)   │ ◄═MCP═► │  (天气服务)        │
│                    │  标准    │                    │
│  不需要知道        │  协议    │  不需要知道        │
│  服务怎么实现的    │         │  谁在调用          │
└────────────────────┘         └────────────────────┘
                               ┌────────────────────┐
            同一个Client ═MCP═► │  MCP Server        │
                               │  (数据库服务)      │
                               └────────────────────┘

MCP 解决了什么：
  ✓ 一次开发，处处可用（不绑定特定框架）
  ✓ 标准化发现（Client自动获取Server提供的工具列表）
  ✓ 标准化调用（统一的输入/输出格式）
  ✓ 独立部署（Server可以是不同语言、不同团队开发的）
```

```python
"""
MCP —— 同一个"查天气"功能，作为独立服务发布
"""

# === MCP Server 端（独立进程/服务） ===

from mcp.server import Server
from mcp.types import Tool, TextContent

server = Server("weather-server")


@server.list_tools()
async def list_tools() -> list[Tool]:
    """声明本Server提供的工具（标准格式）"""
    return [Tool(
        name="get_weather",
        description="获取指定城市的天气信息",
        inputSchema={
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名称"}
            },
            "required": ["city"],
        },
    )]


@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    """处理工具调用请求"""
    if name == "get_weather":
        city = arguments["city"]
        # 实际查询天气...
        return [TextContent(type="text", text=f"{city}: 晴, 25°C")]
    raise ValueError(f"未知工具: {name}")


# === MCP Client 端（在Agent中使用） ===

from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client


async def use_mcp_weather():
    """Agent通过MCP协议调用天气服务"""
    server_params = StdioServerParameters(
        command="python",
        args=["weather_mcp_server.py"],
    )

    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # 发现工具（不需要硬编码工具定义）
            tools = await session.list_tools()
            print(f"可用工具: {[t.name for t in tools.tools]}")

            # 调用工具（标准协议）
            result = await session.call_tool("get_weather", {"city": "北京"})
            print(result.content[0].text)  # "北京: 晴, 25°C"
```

```
小结：
  MCP = 工具的标准化"发布 + 发现 + 调用"协议
  角色 = 连接器 / 标准接口
  核心价值 = 解耦（工具开发者 和 Agent开发者 互不依赖）
  它不是 = 一种新的工具写法，而是工具的分发方式
```

### 2.4 Skill（技能）

```
本质：包含执行策略的可组合能力单元

与 Tool 的区别：
  Tool  = 单步操作（"查天气"）
  Skill = 完整流程（"出行建议" = 查天气 + 查路况 + 生成建议）

与 MCP 的区别：
  MCP   = 工具怎么发布和发现（通信协议）
  Skill = 工具怎么组合和执行（执行策略）

┌──────────────────────────────────────────────────┐
│           Skill: travel_advisor                   │
│                                                  │
│  前置条件: 用户提供了目的地                       │
│                                                  │
│  执行流程:                                       │
│    ① get_weather(city)     ← 调用Tool或MCP       │
│    ② get_traffic(city)     ← 调用Tool或MCP       │
│    ③ LLM综合分析           ← 调用推理模型         │
│    ④ 生成出行建议          ← 格式化输出           │
│                                                  │
│  后置检查: 建议中包含天气和路况信息               │
│  失败回退: 如果天气API挂了，使用缓存数据          │
│                                                  │
│  标签: ["出行", "建议", "综合分析"]               │
│  依赖: [get_weather, get_traffic]                 │
└──────────────────────────────────────────────────┘

Skill 知道三件 Tool 不知道的事：
  1. 何时用（前置条件 + 语义匹配）
  2. 怎么组合（多步骤编排）
  3. 出错怎么办（回退策略）
```

```python
"""
Skill —— 组合多个Tool/MCP为完整能力
"""

from dataclasses import dataclass, field


@dataclass
class SkillStep:
    name: str
    tool: str               # 工具名（可以是本地Tool或MCP工具）
    params_template: dict    # 参数模板（支持引用上一步结果）
    description: str = ""


@dataclass
class Skill:
    name: str
    description: str
    steps: list[SkillStep]
    preconditions: list[str] = field(default_factory=list)
    postconditions: list[str] = field(default_factory=list)
    fallback: str | None = None
    tags: list[str] = field(default_factory=list)


# 定义一个"出行建议"技能
travel_advisor = Skill(
    name="travel_advisor",
    description="根据目的地提供综合出行建议（天气+路况+建议）",
    preconditions=["用户提供了目的地城市"],
    steps=[
        SkillStep("查天气", "get_weather", {"city": "$destination"}),
        SkillStep("查路况", "get_traffic", {"city": "$destination"}),
        SkillStep("生成建议", "llm_generate", {
            "prompt": "根据天气($step_0_result)和路况($step_1_result)，"
                      "为去$destination的出行提供建议"
        }),
    ],
    postconditions=["输出包含天气信息", "输出包含路况信息"],
    fallback="weather_cache_fallback",
    tags=["出行", "建议", "综合分析"],
)

# 对比：同样的功能用 Tool 实现
@tool
def get_travel_advice(city: str) -> str:
    """获取出行建议"""
    weather = get_weather(city)    # 手动调用
    traffic = get_traffic(city)    # 手动调用
    # 手动组合... 没有前置检查、没有回退、没有后置检查
    return f"天气: {weather}, 路况: {traffic}"

# ↑ Tool版本的问题：
# - 如果 get_weather 挂了？→ 整个函数崩溃
# - 如果要换成并行调用？→ 改代码
# - 如果要加缓存回退？→ 改代码
# - 如果LLM想动态是否跳过路况查询？→ 做不到

# Skill版本这些都是声明式的，执行器统一处理
```

## 四、全维度对比表

| 维度 | Function Calling | Tool | MCP | Skill |
|------|-----------------|------|-----|-------|
| **是什么** | LLM的输出格式 | 可执行的功能函数 | 工具分发协议 | 可组合的能力单元 |
| **层级** | 最底层 | 实现层 | 协议层 | 编排层 |
| **谁定义的** | 模型提供商(OpenAI等) | Agent开发者 | Anthropic(标准) | Agent开发者/框架 |
| **执行者** | 无（只生成指令） | 应用代码 | MCP Server | Skill Executor |
| **运行位置** | LLM内部 | Agent进程内 | 独立进程/服务 | Agent进程内 |
| **粒度** | 单次调用 | 单个函数 | 单个工具/资源 | 多步骤流程 |
| **可组合性** | ✗ | ✗ 手动组合 | ✗ 需额外编排 | ✓ 原生支持 |
| **可发现性** | ✗ 需硬编码 | ✗ 需显式注册 | ✓ 自动发现 | ✓ 语义搜索 |
| **错误处理** | ✗ 无 | △ 开发者自写 | △ Server端处理 | ✓ 回退策略 |
| **跨框架** | △ 格式略有差异 | ✗ 框架耦合 | ✓ 标准协议 | △ 取决于实现 |
| **状态感知** | ✗ | ✗ | △ 通过Resources | ✓ 上下文传递 |
| **典型代表** | OpenAI tools参数 | LangChain @tool | MCP Server/Client | Semantic Kernel Skill |
| **学习成本** | 低 | 低 | 中 | 中-高 |

## 五、同一功能的四种实现对比

以"查询订单状态"为例，看四种方式的代码差异：

### 5.1 Function Calling（原始方式）

```python
"""方式1: 纯 Function Calling —— 手动处理全部流程"""

# 你需要手动做5件事：

# ① 手写JSON Schema
tools = [{
    "type": "function",
    "function": {
        "name": "query_order",
        "description": "查询订单状态",
        "parameters": {
            "type": "object",
            "properties": {
                "order_id": {"type": "string", "description": "订单号"}
            },
            "required": ["order_id"]
        }
    }
}]

# ② 定义函数
def query_order(order_id: str) -> dict:
    # 查数据库...
    return {"order_id": order_id, "status": "已发货", "eta": "明天到达"}

# ③ 发送请求
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "我的订单12345到哪了？"}],
    tools=tools,
)

# ④ 手动解析 + 执行
if response.choices[0].message.tool_calls:
    tc = response.choices[0].message.tool_calls[0]
    args = json.loads(tc.function.arguments)
    result = query_order(**args)  # 手动调用

    # ⑤ 手动把结果喂回
    final = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "user", "content": "我的订单12345到哪了？"},
            response.choices[0].message,
            {"role": "tool", "tool_call_id": tc.id, "content": json.dumps(result)},
        ],
        tools=tools,
    )
```

### 5.2 Tool（框架方式）

```python
"""方式2: Tool（LangChain） —— 框架自动处理调用循环"""

from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent


@tool
def query_order(order_id: str) -> dict:
    """查询订单状态

    Args:
        order_id: 订单号，如 "12345"
    """
    return {"order_id": order_id, "status": "已发货", "eta": "明天到达"}

# ↑ 自动生成JSON Schema，自动注入到LLM，自动执行，自动喂回结果
llm = ChatOpenAI(model="gpt-4o")
agent = create_react_agent(llm, tools=[query_order])

result = agent.invoke({"messages": [{"role": "user", "content": "我的订单12345到哪了？"}]})
# 全自动：解析→调函数→喂回→生成回答
```

### 5.3 MCP（标准协议方式）

```python
"""方式3: MCP —— 订单查询作为独立服务"""

# === 服务端（订单团队维护） ===
from mcp.server import Server
from mcp.types import Tool, TextContent
import json

server = Server("order-server")

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [Tool(
        name="query_order",
        description="查询订单状态",
        inputSchema={
            "type": "object",
            "properties": {
                "order_id": {"type": "string", "description": "订单号"}
            },
            "required": ["order_id"],
        },
    )]

@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "query_order":
        result = {"order_id": arguments["order_id"], "status": "已发货", "eta": "明天到达"}
        return [TextContent(type="text", text=json.dumps(result, ensure_ascii=False))]

# === 客户端（Agent团队使用） ===
# Agent只需配置MCP Server的地址
# 框架自动发现工具、注册到LLM、处理调用
# Agent代码中不需要写query_order的任何实现
```

## 六、组合使用：它们不是互斥的

```
实际项目中，四者通常组合使用：

┌──────────────────────────────────────────────────────┐
│                    Agent 系统                         │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │  Skill: customer_service                        │ │
│  │  （客服技能：编排查订单、查退货、转人工）         │ │
│  │                                                 │ │
│  │    ┌────────────────┐  ┌────────────────┐      │ │
│  │    │  MCP Client    │  │  本地 Tools     │      │ │
│  │    │  ↓             │  │  ↓             │      │ │
│  │    │  订单服务(MCP) │  │  format_reply  │      │ │
│  │    │  物流服务(MCP) │  │  send_email    │      │ │
│  │    │  支付服务(MCP) │  │  create_ticket │      │ │
│  │    └────────────────┘  └────────────────┘      │ │
│  │                                                 │ │
│  └─────────────────────────────────────────────────┘ │
│                                                      │
│  底层通信：Function Calling                          │
│  （LLM通过FC选择Skill → Skill编排Tool/MCP执行）     │
│                                                      │
└──────────────────────────────────────────────────────┘

典型分工：
  Function Calling → LLM与外部世界交互的底层通道
  Tool             → 内部能力（格式化、计算、发邮件等）
  MCP              → 外部服务（其他团队/第三方维护的能力）
  Skill            → 业务流程（组合多个Tool和MCP为完整能力）
```

```python

组合使用示例 —— 客服Agent
"""

from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from mcp import ClientSession


class CustomerServiceAgent:
    """客服Agent：融合 Tool + MCP + Skill"""

    def __init__(self):
        self.llm = ChatOpenAI(model="gpt-4o")
        self.mcp_sessions: dict[str, ClientSession] = {}
        self.local_tools: dict[str, callable] = {}
        self.skills: dict[str, Skill] = {}

    async def setup(self):
        """初始化：注册本地Tool + 连接MCP + 加载Skill"""

        # 本地Tool（内部能力）
        self.local_tools["format_reply"] = format_reply
        self.local_tools["create_ticket"] = create_ticket

        # MCP连接（外部服务）
        self.mcp_sessions["order"] = await connect_mcp("order-server")
        self.mcp_sessions["logistics"] = await connect_mcp("logistics-server")

        # 注册Skill（业务流程）
        self.skills["order_tracking"] = order_tracking
        self.skills["refund_process"] = refund_process

    async def handle(self, query: str) -> str:
        """
        处理用户请求：
        1. LLM通过Function Calling选择使用哪个Skill
        2. Skill Executor编排执行多个步骤
        3. 每个步骤可能调用本地Tool或MCP Server
        4. 返回最终结果

        # 将所有能力（Tool + MCP工具 + Skill）汇总为Function Calling格式
        available_functions = (
            self._tools_as_functions()
            + await self._mcp_as_functions()
            + self._skills_as_functions()
        )

        # LLM决定使用哪个
        response = self.llm.invoke(
            query,
            tools=available_functions,
        )
```

## 七、选型决策指南

```
你应该用哪个？

问题1: 你是在做什么？
  ├─ 直接调OpenAI API，不用框架
  │  → 直接用 Function Calling
  │
  ├─ 在框架(LangChain/CrewAI等)中开发Agent
  │  → 用框架的 Tool 机制
  │
  ├─ 要接入其他团队/第三方的服务
  │  → 接入 MCP Server
  │
  └─ 要编排多步骤的复杂业务流程
     → 设计成 Skill

问题2: 能力是谁维护的？
  ├─ 我自己维护（同一个代码库）
  │  → Tool
  │
  ├─ 其他团队维护 / 第三方服务
  │  → MCP
  │
  └─ 需要动态组合多个能力
     → Skill

问题3: 项目阶段？
  ├─ 快速原型 / PoC
  │  → Tool（最快）
  │
  ├─ 多团队协作 / 产品化
  │  → MCP + Skill
  │
  └─ 企业级生产环境
     → 全部组合使用
```

## 八、常见误解

```
误解1: "MCP会替代Function Calling"
  ✗ 错误。MCP底层仍然依赖Function Calling让LLM发起调用指令。
  ✓ MCP替代的是"每个框架自己定义工具格式"的碎片化现状。

误解2: "Skill就是多个Tool放在一起"
  ✗ 错误。Tool的简单组合只是过程式代码。
  ✓ Skill是声明式的，包含前置条件、编排策略、回退机制、后置验证。

误解3: "用了MCP就不需要Tool了"
  ✗ 错误。不是所有功能都需要独立服务化。
  ✓ 简单的内部功能（格式化、计算）用Tool更合适，外部服务才用MCP。

误解4: "Function Calling是OpenAI专有的"
  ✗ 错误。Claude、Gemini、Mistral、国产模型都支持。
  ✓ 各家格式略有差异，但核心机制相同，框架(LangChain等)屏蔽了差异。

误解5: "这四个概念要选一个用"
  ✗ 错误。它们是不同层级的抽象。
  ✓ 生产系统通常四者组合使用，各司其职。
```
