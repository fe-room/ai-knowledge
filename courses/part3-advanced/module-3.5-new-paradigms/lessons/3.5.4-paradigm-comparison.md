## 一、三大范式总览

### 1.1 范式概况

```
2025年Agent开发的三条路径：

路径A：自建Agent
┌───────────────────────────────────────────┐
│  自己写Agent循环 + 自己集成工具 + 自己管理状态 │
│  框架支撑：LangChain / LangGraph / CrewAI  │
│  控制力：★★★★★   开发量：★★★★★            │
└───────────────────────────────────────────┘

路径B：Assistants API（托管Agent）
┌───────────────────────────────────────────┐
│  平台托管Agent状态 + 内置工具 + 简化调用   │
│  平台方：OpenAI Assistants API             │
│  控制力：★★☆☆☆   开发量：★★☆☆☆            │
└───────────────────────────────────────────┘

路径C：MCP标准化
┌───────────────────────────────────────────┐
│  Agent逻辑自建/托管 + 工具层标准化         │
│  工具发现和调用走MCP协议                    │
│  控制力：★★★★☆   开发量：★★★☆☆            │
└───────────────────────────────────────────┘

注意：这三者不是互斥关系，可以组合使用
```

## 二、选型决策框架

### 2.1 决策树

```
开始
  │
  ├─ 是否需要快速验证想法（MVP/原型）？
  │   ├─ 是 → ★ Assistants API
  │   └─ 否 ↓
  │
  ├─ 数据隐私是否为硬性要求？
  │   ├─ 是（医疗/金融/政务） → ★ 自建Agent
  │   └─ 否 ↓
  │
  ├─ 是否需要多模型支持？
  │   ├─ 是（OpenAI + Claude + 开源） → ★ 自建Agent
  │   └─ 否 ↓
  │
  ├─ 是否有大量工具需要跨应用共享？
  │   ├─ 是 → ★ MCP + 自建/Assistants
  │   └─ 否 ↓
  │
  ├─ 团队的AI工程经验如何？
  │   ├─ 新手团队 → ★ Assistants API（起步） → 逐步自建
  │   ├─ 有经验 → ★ 自建Agent（主） + MCP（工具层）
  │   └─ 资深团队 → ★ 混合架构
  │
  └─ 默认推荐 → 自建Agent + MCP工具层
```

## 三、混合架构设计

### 3.1 推荐的混合架构

```
最适合生产环境的混合方案：

┌─────────────────────────────────────────────────────────┐
│                    应用层                                │
│     ┌──────────────────────────────────────┐            │
│     │          Agent编排层                  │            │
│     │   LangGraph / 自建状态机              │            │
│     │                                      │            │
│     │   ┌──────────┐  ┌──────────┐        │            │
│     │   │ 推理模型  │  │ 快速模型  │        │            │
│     │   │(规划/验证)│  │(分类/生成)│        │            │
│     │   └──────────┘  └──────────┘        │            │
│     └──────────────┬───────────────────────┘            │
│                    │                                    │
│     ┌──────────────↓───────────────────────┐            │
│     │          MCP 工具层                   │            │
│     │   ┌───────┐ ┌───────┐ ┌───────┐     │            │
│     │   │Server │ │Server │ │Server │     │            │
│     │   │(数据库)│ │(搜索) │ │(文件) │     │            │
│     │   └───────┘ └───────┘ └───────┘     │            │
│     └──────────────────────────────────────┘            │
│                                                         │
└─────────────────────────────────────────────────────────┘

优势：
  → Agent逻辑自主可控（不依赖单一平台）
  → 工具标准化（MCP协议，任何Client可复用）
  → 模型灵活切换（推理模型 + 快速模型分层）
  → 每一层可独立替换和升级
```

### 3.2 实现混合架构

```python
"""
混合架构示例：LangGraph Agent + MCP工具 + 推理模型分层
"""

from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
from openai import OpenAI
import operator

client = OpenAI()

# ===== 状态定义 =====

class AgentState(TypedDict):
    query: str
    intent: str
    plan: list[dict]
    tool_results: Annotated[list[str], operator.add]
    answer: str

# ===== 节点函数 =====

def classify_intent(state: AgentState) -> dict:
    """意图分类 — 用快速模型"""
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{
            "role": "user",
            "content": f"将以下请求分类（simple/complex/creative）：{state['query']}"
        }]
    )
    return {"intent": response.choices[0].message.content.strip()}


def create_plan(state: AgentState) -> dict:
    """任务规划 — 复杂任务用推理模型"""
    if "complex" in state["intent"]:
        response = client.chat.completions.create(
            model="o3-mini",
            reasoning_effort="high",
            messages=[{
                "role": "user",
                "content": f"制定执行计划（JSON数组，每步含action和tool字段）：{state['query']}"
            }]
        )
    else:
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{
                "role": "user",
                "content": f"简要回答即可：{state['query']}"
            }]
        )

    import json
    try:
        plan = json.loads(response.choices[0].message.content)
    except json.JSONDecodeError:
        plan = [{"action": "direct_answer", "tool": "none"}]

    return {"plan": plan}


async def execute_tools(state: AgentState) -> dict:
    """执行工具 — 通过MCP调用"""
    results = []
    for step in state["plan"]:
        tool_name = step.get("tool", "none")
        if tool_name != "none":
            # 通过MCP Client调用对应Server的工具
            # result = await mcp_agent.call_tool(tool_name, step.get("input", {}))
            result = f"[{tool_name}执行结果]"  # 示例占位
            results.append(result)
    return {"tool_results": results}


def generate_answer(state: AgentState) -> dict:
    """生成最终回答 — 标准模型"""
    context = "\n".join(state.get("tool_results", []))
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": "基于工具执行结果回答用户问题。"},
            {"role": "user", "content": f"问题：{state['query']}\n\n工具结果：{context}"}
        ]
    )
    return {"answer": response.choices[0].message.content}


# ===== 构建图 =====

def should_use_tools(state: AgentState) -> str:
    """路由：是否需要工具调用"""
    if state["plan"] and state["plan"][0].get("tool") != "none":
        return "tools"
    return "answer"


graph = StateGraph(AgentState)
graph.add_node("classify", classify_intent)
graph.add_node("plan", create_plan)
graph.add_node("tools", execute_tools)
graph.add_node("answer", generate_answer)

graph.set_entry_point("classify")
graph.add_edge("classify", "plan")
graph.add_conditional_edges("plan", should_use_tools, {
    "tools": "tools",
    "answer": "answer",
})
graph.add_edge("tools", "answer")
graph.add_edge("answer", END)

app = graph.compile()
```

## 四、演进路线与学习建议

### 4.1 技术演进趋势

```
2025及以后的趋势判断：

1. MCP将成为主流工具标准
   → Anthropic发起，已获得广泛生态支持
   → 类比：REST API之于Web服务
   → 学习优先级：⭐⭐⭐⭐⭐

2. 推理模型从专用走向通用
   → 推理能力正在被整合进通用模型
   → 未来可能模型自动判断何时需要深度推理
   → 学习优先级：⭐⭐⭐⭐

3. Assistants API等托管方案持续优化
   → 更适合中小团队和快速原型
   → 但不会替代自建方案
   → 学习优先级：⭐⭐⭐

4. Agent框架趋向标准化
   → LangGraph / 状态机模式正在成为共识
   → 框架之间的差异在缩小

5. 端侧推理能力增强
   → 小模型在本地运行Agent
   → WebLLM / Ollama + Agent的组合
```

### 4.2 学习路径建议

```
阶段1：打好基础（你已完成模块2.3-2.5）
  → 理解Agent循环、Function Calling、RAG

阶段2：了解新范式（本模块）
  ├─ 1. 先学MCP（3.5.1）→ 改变你的工具集成方式
  ├─ 2. 体验Assistants API（3.5.2）→ 了解托管方案的能力边界
  ├─ 3. 掌握推理模型（3.5.3）→ 学会在Agent中分层使用
  └─ 4. 建立选型判断力（3.5.4）→ 本节

阶段3：实战混合架构
  → 用自建Agent + MCP + 推理模型重构项目3或项目4
  → 对比改造前后的开发效率和系统能力

阶段4：持续关注
  → MCP生态（新的Server实现、新的Client支持）
  → 新推理模型发布（能力和成本的变化）
  → Agent框架的演进（LangGraph新特性等）
```

## 五、常见误区

```
误区1："新范式出来了，以前学的都没用了"
  → 错。MCP是工具集成层的升级，Agent核心逻辑不变。
  → ReAct、Plan-Execute等模式仍然是基础。

误区2："Assistants API可以替代一切自建方案"
  → 错。多模型、高隐私、精细控制场景仍然需要自建。
  → Assistants API是锦上添花，不是银弹。

误区3："推理模型比普通模型好，应该全部使用"
  → 错。推理模型的延迟和成本远高于普通模型。
  → 只在关键决策点使用（规划、验证），其余用快速模型。

误区4："MCP会完全替代Function Calling"
  → 不完全。MCP是工具发现和调用的标准协议，
  → Function Calling仍然是LLM与工具交互的底层机制。
  → MCP是更高层的抽象，两者是互补关系。

误区5："选最新的技术就对了"
  → 错。合适的技术 > 最新的技术。
  → 简单的FAQ机器人用LangChain + 几个工具就够了，
  → 不需要上MCP + Assistants + 推理模型的全家桶。
```
