# 3.5.3 推理模型与深度思考

> 理解推理模型如何改变Agent的决策能力

## 一、从生成到推理

### 1.1 传统生成模型的局限

```
传统Chat模型（GPT-4o / Claude Sonnet等）：

用户问题 → 模型 → 逐token生成回答

特点：
  ✅ 速度快（流式输出，即时响应）
  ✅ 成本低
  ✅ 适合日常对话、文本生成、简单推理

  ❌ 复杂推理容易出错
  ❌ 多步骤问题易丢失逻辑链
  ❌ 数学和逻辑题准确率有限

示例：
  问："一个房间里有3个杀手，一个人进去杀了1个，房间里还有几个人？"
  传统模型可能回答"2个"（错误 — 忽略了进来的人和被杀的尸体）
```

### 1.2 推理模型的突破

```
推理模型（o1 / o3 / Claude Extended Thinking）：

用户问题 → 模型内部深度思考 → 输出最终回答

┌──────────────────────────────────────────────┐
│           模型内部（用户不可见）               │
│                                              │
│   思考步骤1：分析问题...                      │
│   思考步骤2：考虑边界情况...                  │
│   思考步骤3：验证逻辑...                      │
│   思考步骤4：修正错误...                      │
│   思考步骤5：得出结论...                      │
│                                              │
│   → 内部消耗大量thinking tokens              │
│   → 外部只展示最终答案                        │
└──────────────────────────────────────────────┘

特点：
  ✅ 复杂推理能力大幅提升
  ✅ 数学、编程、逻辑题准确率显著提高
  ✅ 能自我纠错和验证

  ❌ 延迟高（需要思考时间，数秒到数十秒）
  ❌ 成本高（thinking tokens也收费）
  ❌ 不适合简单对话场景
```

### 1.3 主流推理模型对比

| 模型 | 提供商 | 特点 | 适用场景 |
|------|--------|------|----------|
| **o1** | OpenAI | 首个推理模型，通用能力强 | 复杂分析、战略规划 |
| **o3-mini** | OpenAI | 轻量级，可调推理深度 | 性价比场景，可控成本 |
| **o3** | OpenAI | 最强推理能力 | 科研级复杂问题 |
| **Claude Extended Thinking** | Anthropic | Claude系列推理模式 | 长逻辑链、代码分析 |
| **Gemini 2.0 Flash Thinking** | Google | 速度和思考的平衡 | 快速推理场景 |
| **DeepSeek-R1** | DeepSeek | 开源推理模型 | 自部署、隐私敏感 |

## 二、推理模型API使用

### 2.1 OpenAI o3-mini

```python
"""
OpenAI 推理模型使用
"""

from openai import OpenAI

client = OpenAI()

# ===== 基础用法 =====

response = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="medium",  # low / medium / high
    messages=[
        {
            "role": "user",
            "content": """
            设计一个分布式系统来处理每秒100万条日志消息。
            要求：低延迟写入、支持实时查询、数据保留30天。
            请给出完整的技术方案。
            """
        }
    ]
)

print(response.choices[0].message.content)

# 输出中包含完整的推理流程：
# - 分析流量需求
# - 选择存储方案（如Kafka + ClickHouse）
# - 设计分区策略
# - 考虑故障恢复
# - 估算硬件成本
```

### 2.2 reasoning_effort 参数

```python
"""
推理深度控制 —— 在质量和成本间找平衡
"""

# low：快速回答，适合简单推理
response_low = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="low",
    messages=[{"role": "user", "content": "123 × 456 = ?"}]
)
# 消耗少量 thinking tokens，快速出结果

# medium：中等推理，适合中等复杂度
response_med = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="medium",
    messages=[{"role": "user", "content": "分析这段代码的时间复杂度并优化"}]
)

# high：深度推理，适合高复杂度问题
response_high = client.chat.completions.create(
    model="o3-mini",
    reasoning_effort="high",
    messages=[{"role": "user", "content": "证明P ≠ NP问题的已知进展和关键困难"}]
)

# Token消耗对比（示例）：
# low:    ~500 thinking tokens  + ~200 output tokens
# medium: ~2000 thinking tokens + ~500 output tokens
# high:   ~8000 thinking tokens + ~1000 output tokens
```

### 2.3 Claude Extended Thinking

```python
"""
Claude 扩展思考模式
"""

import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000,  # 思考预算：允许消耗的最大思考token数
    },
    messages=[
        {
            "role": "user",
            "content": """
            审查以下Python代码的安全问题，考虑：
            1. 注入攻击
            2. 认证绕过
            3. 数据泄露
            4. 竞态条件

            ```python
            @app.route('/api/users/<user_id>')
            def get_user(user_id):
                query = f"SELECT * FROM users WHERE id = {user_id}"
                result = db.execute(query)
                return jsonify(result)
            ```
            """
        }
    ]
)

# 读取思考过程和最终回答
for block in response.content:
    if block.type == "thinking":
        print("=== 思考过程 ===")
        print(block.thinking)  # 可以看到完整的推理链
    elif block.type == "text":
        print("=== 最终回答 ===")
        print(block.text)
```

### 2.4 重要使用限制

```
⚠️ 推理模型的API差异（与普通Chat模型不同）：

OpenAI o系列：
  ❌ 不支持 system message（使用 developer message 代替）
  ❌ 不支持 temperature 参数（固定为1）
  ❌ 不支持 top_p / frequency_penalty 等采样参数
  ❌ 不支持 streaming（o1时代；o3已部分支持）
  ✅ 支持 function calling / tool use
  ✅ 支持 structured outputs

Claude Extended Thinking：
  ❌ 不支持 temperature（必须为1）
  ❌ thinking blocks 不能在后续对话中传回
  ✅ 支持 streaming
  ✅ 支持 tool use

通用注意：
  → 推理模型的输出不完全确定性（即使相同输入，多次调用结果可能不同）
  → system prompt / instructions 仍然有效，但需要更简洁
```

## 三、推理模型在Agent中的应用

### 3.1 分层模型策略

```
实际Agent系统中，不必所有环节都用推理模型

最佳实践 —— 分层使用：
┌─────────────────────────────────────────────────┐
│                                                 │
│  用户请求                                        │
│      ↓                                          │
│  ┌──────────────┐                               │
│  │ 意图分类      │ ← GPT-4o-mini（快速、便宜）   │
│  └──────┬───────┘                               │
│         ↓                                       │
│  ┌──────────────┐                               │
│  │ 任务规划      │ ← o3-mini/high（需要深度推理）│
│  └──────┬───────┘                               │
│         ↓                                       │
│  ┌──────────────┐                               │
│  │ 工具调用      │ ← GPT-4o（平衡速度和能力）    │
│  └──────┬───────┘                               │
│         ↓                                       │
│  ┌──────────────┐                               │
│  │ 回答生成      │ ← GPT-4o（流式输出，快速）    │
│  └──────┬───────┘                               │
│         ↓                                       │
│  ┌──────────────┐                               │
│  │ 质量检查      │ ← o3-mini/low（快速验证）     │
│  └──────────────┘                               │
│                                                 │
└─────────────────────────────────────────────────┘

原则：只在需要深度推理的关键环节使用推理模型
```

### 3.2 实现分层Agent

```python
"""
分层模型Agent —— 关键决策用推理模型，其他用快速模型
"""

from openai import OpenAI

client = OpenAI()

class TieredAgent:
    """分层模型Agent"""

    # 模型路由配置
    MODEL_CONFIG = {
        "classify": {"model": "gpt-4o-mini", "desc": "意图分类（快速）"},
        "plan":     {"model": "o3-mini",      "desc": "任务规划（深度推理）"},
        "execute":  {"model": "gpt-4o",       "desc": "工具调用（平衡）"},
        "generate": {"model": "gpt-4o",       "desc": "文本生成（流式）"},
        "verify":   {"model": "o3-mini",      "desc": "质量验证（推理）"},
    }

    def __init__(self):
        self.tools = {}

    def _call_llm(self, stage: str, messages: list, **kwargs) -> str:
        """根据阶段选择模型"""
        config = self.MODEL_CONFIG[stage]
        model = config["model"]

        params = {"model": model, "messages": messages}

        # 推理模型使用 reasoning_effort
        if model.startswith("o3"):
            effort = kwargs.get("reasoning_effort", "medium")
            params["reasoning_effort"] = effort
        else:
            params["temperature"] = kwargs.get("temperature", 0.7)

        response = client.chat.completions.create(**params)
        return response.choices[0].message.content

    def run(self, user_query: str) -> str:
        """完整的分层执行流程"""

        # 阶段1：意图分类（快速模型）
        intent = self._call_llm("classify", [
            {"role": "system", "content": "将用户请求分类为: simple_qa / complex_analysis / creative_writing / code_task"},
            {"role": "user", "content": user_query}
        ])

        # 阶段2：任务规划（推理模型，仅复杂任务）
        if "complex" in intent or "code" in intent:
            plan = self._call_llm("plan", [
                {"role": "user", "content": f"请制定详细的执行计划：\n{user_query}"}
            ], reasoning_effort="high")
        else:
            plan = user_query  # 简单任务跳过规划

        # 阶段3：执行（标准模型）
        result = self._call_llm("generate", [
            {"role": "system", "content": "按照计划执行任务并输出结果。"},
            {"role": "user", "content": f"计划：{plan}\n\n原始问题：{user_query}"}
        ])

        # 阶段4：验证（推理模型，低effort快速检查）
        check = self._call_llm("verify", [
            {"role": "user", "content": f"检查这个回答是否正确和完整：\n问题：{user_query}\n回答：{result}\n如果有问题请指出，没问题回复OK"}
        ], reasoning_effort="low")

        if "OK" not in check:
            # 有问题，用推理模型重新生成
            result = self._call_llm("plan", [
                {"role": "user", "content": f"原始问题：{user_query}\n初次回答有问题：{check}\n请重新给出正确的回答。"}
            ], reasoning_effort="high")

        return result
```

### 3.3 推理模型用于Plan-and-Execute

```python
"""
将推理模型用在Agent的"规划"阶段
—— 推理模型最适合的Agent环节
"""

import json

def plan_with_reasoning(client, task: str, available_tools: list[dict]) -> list[dict]:
    """使用推理模型做任务规划"""

    tools_desc = "\n".join(
        f"- {t['name']}: {t['description']}"
        for t in available_tools
    )

    response = client.chat.completions.create(
        model="o3-mini",
        reasoning_effort="high",
        response_format={"type": "json_object"},
        messages=[{
            "role": "user",
            "content": f"""你是一个任务规划器。请分析任务并制定执行计划。

任务：{task}

可用工具：
{tools_desc}

请输出JSON格式的执行计划：
{{
  "analysis": "任务分析",
  "steps": [
    {{
      "step": 1,
      "action": "使用的工具或操作",
      "input": "输入参数",
      "depends_on": [],
      "reason": "为什么需要这一步"
    }}
  ],
  "risks": ["可能的风险"],
  "fallback": "备选方案"
}}"""
        }]
    )

    plan = json.loads(response.choices[0].message.content)
    return plan


# 使用示例
plan = plan_with_reasoning(
    client,
    task="调研竞品A和竞品B的最新功能更新，分析差异，生成对比报告",
    available_tools=[
        {"name": "web_search", "description": "搜索互联网"},
        {"name": "read_url", "description": "读取网页内容"},
        {"name": "write_doc", "description": "生成文档"},
    ]
)
# 推理模型会产出周密的执行计划，包含依赖关系和风险评估
```

## 四、成本与延迟权衡

### 4.1 成本对比

```
模型                  输入价格        输出价格        思考价格
─────────────────────────────────────────────────────────────
gpt-4o-mini          $0.15/1M       $0.60/1M        N/A
gpt-4o               $2.50/1M       $10.00/1M       N/A
o3-mini (low)        $1.10/1M       $4.40/1M        含在输出中
o3-mini (high)       $1.10/1M       $4.40/1M        含在输出中*
o3                   $10.00/1M      $40.00/1M       含在输出中
claude-sonnet        $3.00/1M       $15.00/1M       $3.00/1M

* high effort 会产生更多 thinking tokens，总成本更高

示例计算（一次复杂分析任务）：
  输入: 2000 tokens, 输出: 1000 tokens

  gpt-4o-mini: 2000×$0.15/1M + 1000×$0.60/1M ≈ $0.001
  gpt-4o:      2000×$2.50/1M + 1000×$10.0/1M ≈ $0.015
  o3-mini:     2000×$1.10/1M + 5000×$4.40/1M ≈ $0.024  (含~4000 thinking tokens)
  o3:          2000×$10.0/1M + 20000×$40/1M  ≈ $0.820  (含~19000 thinking tokens)

  → 推理模型在高effort下成本可达普通模型的 50-80倍
```

### 4.2 何时值得用推理模型

```
✅ 值得使用推理模型的场景：

  1. 关键决策点
     → Agent的任务规划 / 方案选择
     → 错误决策的代价远高于模型成本

  2. 复杂代码生成与审查
     → 需要考虑边界条件、安全漏洞
     → 多文件修改的一致性检查

  3. 数据分析与推理
     → 多因素分析
     → 需要逻辑推导的场景

  4. 质量关卡
     → 最终输出的质量验证
     → 但用 low effort 控制成本

❌ 不值得使用的场景：

  1. 简单对话 / FAQ回答
  2. 文本翻译、摘要
  3. 格式化输出
  4. 高频调用的分类/提取任务（成本不可控）
```

## 五、Structured Outputs（附录）

### 5.1 为什么要提Structured Outputs

```
与推理模型配合的关键技术 ——
确保模型输出严格符合JSON Schema

传统方式：
  → prompt中写"请输出JSON"
  → 模型可能输出不合法的JSON
  → 需要try/except + 重试

Structured Outputs：
  → API层面强制保证输出格式
  → 100%合规（基于受约束解码）
  → 对Agent特别重要：工具调用参数必须可靠
```

### 5.2 使用方法

```python
"""
Structured Outputs —— 强制JSON Schema合规
"""

from pydantic import BaseModel

# 方式1：response_format + json_schema

response = client.chat.completions.create(
    model="gpt-4o",
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "task_plan",
            "strict": True,  # 严格模式
            "schema": {
                "type": "object",
                "properties": {
                    "steps": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "action": {"type": "string"},
                                "tool": {"type": "string"},
                            },
                            "required": ["action", "tool"],
                            "additionalProperties": False,
                        }
                    },
                },
                "required": ["steps"],
                "additionalProperties": False,
            }
        }
    },
    messages=[{"role": "user", "content": "制定一个数据分析计划"}]
)

# 100%保证输出符合上述schema，无需try/except


# 方式2：使用 Pydantic 模型（更简洁）

class Step(BaseModel):
    action: str
    tool: str
    reason: str

class TaskPlan(BaseModel):
    steps: list[Step]
    total_time_estimate: str

response = client.beta.chat.completions.parse(
    model="gpt-4o",
    response_format=TaskPlan,
    messages=[{"role": "user", "content": "制定一个数据分析计划"}]
)

plan = response.choices[0].message.parsed  # 直接得到TaskPlan对象
for step in plan.steps:
    print(f"{step.action} → 使用 {step.tool}（{step.reason}）")
```

## 六、小结

```
推理模型要点回顾：

┌───────────────────────────────────────────────┐
│  1. 核心区别                                   │
│     → 传统模型：即时生成，快而浅               │
│     → 推理模型：先想后答，慢而深               │
│                                               │
│  2. 实用策略                                   │
│     → 分层使用：关键环节推理模型，其余快速模型 │
│     → reasoning_effort：按需调整推理深度       │
│     → 不要无脑全用推理模型（成本和延迟不值得） │
│                                               │
│  3. Agent中的最佳位置                          │
│     → 任务规划（Plan阶段）                     │
│     → 方案选择（多路径决策）                   │
│     → 质量验证（最终检查）                     │
│                                               │
│  4. 搭配Structured Outputs                    │
│     → 确保推理模型的输出可被程序可靠解析       │
│     → Agent工具调用、计划生成都需要结构化输出  │
│                                               │
│  5. 关注开源                                   │
│     → DeepSeek-R1 等开源推理模型               │
│     → 自部署场景的隐私和成本优势               │
└───────────────────────────────────────────────┘
```
